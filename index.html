Slide 03

¿Por qué Clean Architecture + DDD?

No es moda. Cada patrón resuelve un problema concreto.

🧱 Clean Architecture separación

Si mañana cambiamos de MariaDB a PostgreSQL, solo tocamos Infrastructure. El Domain, Application y Web no se enteran. Sin Clean Architecture, el cambio de BD rompe medio sistema.

Lo mismo si cambiamos Blazor por React: el API y toda la lógica de negocio sobreviven intactos.

📐 DDD — Domain-Driven Design lenguaje

El código habla el mismo idioma que el negocio. En EsSalud existe MicroRed, Ipress, Serie. En el código también. No hay traducción mental.

Las reglas de negocio viven en el Domain: if (numeroFin <= numeroInicio) throw. No en el controller, no en el frontend.

🔧 Feature Slicing escalabilidad

Todo lo de MicroRed vive junto:
Application/Ubicaciones/MicroRedes/
DTOs, interfaces, UseCases — sin buscar en 5 carpetas distintas.

Cuando agregan un practicante nuevo, puede trabajar en su feature sin tocar la de otro.

📦 Un UseCase por operación claridad

Cada clase tiene una sola razón para cambiar. CrearMicroRedUseCase solo crea. DesactivarMicroRedUseCase solo desactiva. Si hay un bug en la desactivación, sabes exactamente dónde ir.

Sin este patrón: un método gigante en el controller que hace todo y nadie entiende qué puede romper.

Slide 04

El Domain — la capa más importante

No depende de nada. No conoce EF Core, ni HTTP, ni Blazor. Solo negocio puro.

AuditableEntity (Shared)

Clase base que todas las entidades heredan. Provee automáticamente:

  • TenantId — multi-empresa
  • FechaCreacion / CreadoPor
  • FechaActualizacion / ActualizadoPor
  • Eliminado / FechaEliminacion (soft delete)
  • ActualizarFecha(usuarioId)
  • MarcarEliminado(usuarioId)

Responsabilidad del Domain

Una entidad Domain solo hace tres cosas:

  • Validar sus propios datos en el constructor
  • Mutar su propio estado via métodos
  • Lanzar excepción si algo es inválido

🔑 NO llama ActualizarFecha() internamente. Eso lo hace el UseCase, que es quien sabe el usuarioId.

Separación de acciones

Cada entidad tiene dos operaciones de estado separadas:

  • CambiarEstado() → toggle activo/inactivo, sigue visible
  • MarcarEliminado() → soft delete real, desaparece

Dos endpoints distintos: PATCH /estado y DELETE

Serie.cs — ejemplo completo del patrón correcto
// ✅ Constructor valida reglas de negocio
public Serie(int microRedId, int anio, int numeroInicio, int numeroFin)
{
    if (microRedId <= 0)           throw new ArgumentException("La MicroRed es obligatoria.");
    if (numeroFin <= numeroInicio)  throw new ArgumentException("Fin debe ser mayor al inicio.");
    MicroRedId = microRedId; Anio = anio; Activo = true;
}

// ✅ Mutación pura — sin lógica de auditoría (eso lo hace el UseCase)
public void Actualizar(int anio, int numeroInicio, int numeroFin)
{
    if (numeroFin <= numeroInicio) throw new ArgumentException("Fin debe ser mayor al inicio.");
    Anio = anio; NumeroInicio = numeroInicio; NumeroFin = numeroFin;
    // ← NO hay ActualizarFecha() aquí. El UseCase la llama con el userId.
}
Slide 05

Application — los casos de uso

Orquesta el dominio. Sabe qué pasa, en qué orden, con quién.

EditarSerieUseCase — patrón correcto ✅
public async Task<Result> EjecutarAsync(
    int id, UpdateSerieDto dto,
    int actualizadoPor, int tenantId)
{
    var entity = await repo.GetByIdAsync(id, tenantId);
    if (entity is null) return Result.Failure("No encontrado.");
    try {
        entity.Actualizar(dto.Anio, dto.NumeroInicio, dto.NumeroFin);
        entity.ActualizarFecha(actualizadoPor); // ← con userId
        await repo.UpdateAsync(entity);
        return Result.Success();
    }
    catch (ArgumentException ex) { return Result.Failure(ex.Message); }
}
¿Por qué Result<T> en lugar de exceptions?

El UseCase devuelve un resultado tipado: Result.Success() o Result.Failure("mensaje"). El Controller decide el código HTTP. Así el dominio no depende de HTTP ni de cómo se entrega el error.

5 UseCases por entidad — cada uno con un único propósito:
CrearXxxUseCase POST — nuevo registro
ListarXxxUseCase GET — todos los registros
EditarXxxUseCase PUT — actualizar datos
DesactivarXxxUseCase PATCH /estado — toggle
EliminarXxxUseCase DELETE — soft delete real
IRepository vive en Application ← importante
La interfaz ISerieRepository está en Application. La implementación concreta SerieRepository está en Infrastructure. Application no sabe que existe EF Core.
Slide 06

¿Por qué constructores distintos?

Controllers usan constructor clásico. UseCases y ApiServices usan primary constructor (C# 12). No es al azar.

Controller — Clásico complejidad alta

❌ Primary (no usar)
public class MicroRedController( CrearMicroRedUseCase crear, ListarMicroRedUseCase listar, EditarMicroRedUseCase editar, DesactivarMicroRedUseCase des, EliminarMicroRedUseCase eli, CrearSerieUseCase cSerie, ...más) : ControllerBase
✅ Clásico (usar)
public class MicroRedController : ControllerBase { private readonly CrearMicroRedUseCase _crear; ... public MicroRedController( CrearMicroRedUseCase crear, ...) { _crear = crear; } }

Con muchas dependencias (8-10 en MicroRedController con Series), el primary constructor en la firma de clase se hace ilegible. El clásico es más explícito y más fácil de debuggear.

UseCase — Primary simplicidad

❌ Clásico (innecesario)
public class CrearSerieUseCase { private readonly ISerieRepository _repo; public CrearSerieUseCase( ISerieRepository repo) { _repo = repo; } public async Task EjecutarAsync... }
✅ Primary (usar)
public class CrearSerieUseCase( ISerieRepository repo) { public async Task EjecutarAsync( CreateSerieDto dto, int creadoPor, int tenantId) { var entity = new Serie(...); ... } }

Un UseCase tiene 1-2 dependencias máximo. El primary constructor elimina 6 líneas de boilerplate sin perder claridad. El compilador genera exactamente el mismo código.

📌 Regla del proyecto: Controllers y Repositories → constructor clásico. UseCases y ApiServices Web → primary constructor. Nunca mezclar estilos dentro del mismo tipo de clase.
Slide 07

JWT puro — sin ASP.NET Core Identity

Elegimos JWT propio porque tenemos multi-tenancy y roles específicos del negocio.

1

Usuario hace login

POST api/auth/login con DNI + password. El LoginUseCase verifica contra la BD.

2

API devuelve token JWT

El token incluye: userId, tenantId, rol, expiración. Todo firmado con la clave secreta.

3

Web guarda el token

El Blazor TokenService lo guarda en memoria de sesión. Cada petición lo envía en el header Authorization: Bearer xxx.

4

API valida automáticamente

El middleware de ASP.NET verifica la firma del token. Si falla → 401. Si el rol no alcanza → 403. Sin tocar la BD.

5

Controller extrae datos

ClaimsHelper.ObtenerDatos(User) extrae userId y tenantId del token para pasarlos al UseCase.

¿Por qué NO Identity?

ASP.NET Core Identity asume un modelo de usuario fijo con tablas propias. Nosotros tenemos multi-tenant (una BD para N empresas), roles del negocio propios (Operador, Supervisor, Lectura) y la tabla usuario vive en Shared. Identity pelea con todo eso. JWT puro nos da control total.

Policies de autorización
SoloAdmins→ SuperAdmin, Admin
AccesoOT→ + Operador
Supervisores→ + Supervisor
SoloLectura→ + Lectura

Ubicaciones y MasterData usan [Authorize(Policy = Policies.SoloAdmins)]. Los controllers de operación usan AccesoOT.

Slide 08

Reglas que no se rompen

Estas convenciones mantienen el proyecto coherente cuando trabajan muchas personas.

Regla Por qué Ejemplo
Una clase por archivo Git muestra cambios por archivo. Con dos clases en uno, un cambio toca un archivo que parece no relacionado. MicroRed.cs solo tiene MicroRed
Namespaces file-scoped Menos indentación, más código visible en pantalla. C# 10+ lo permite nativamente. namespace Xxx.Yyy; sin llaves
IRepository en Application Application no puede depender de Infrastructure. La interfaz actúa como contrato que Infrastructure implementa. ISerieRepository en App, SerieRepository en Infra
Query filter automático Cada consulta filtra automáticamente por TenantId y Eliminado=false. Imposible mostrar datos de otra empresa por error. .HasQueryFilter(x => !x.Eliminado && x.TenantId == ...)
ActualizarFecha() en UseCase El Domain no conoce quién es el usuario. Solo el UseCase tiene acceso a userId del token. entity.Actualizar(...); entity.ActualizarFecha(userId);
PATCH /estado ≠ DELETE Desactivar (toggle) y eliminar son acciones distintas. Desactivar mantiene el dato visible como inactivo; eliminar lo borra de la lista. CambiarEstado() vs MarcarEliminado()
GetAsync antes de ReadFromJsonAsync Así podemos interceptar 401 y 403 antes de intentar deserializar. Con GetFromJsonAsync el error se pierde. var r = await http.GetAsync(url);
